# loop-constraints.md — what the agent must NEVER do

This file is the structural counterpart to `docs/safety.md`. Where
`safety.md` enforces human gates on destructive *operations* (git push,
issue close, …), this file enforces denylists on *paths* and *op kinds*
the agent must not mutate.

The two policy docs overlap on the operations side on purpose — both must
agree before a change goes through. If `safety.md` and `constraints.md`
disagree, **the more restrictive one wins**.

## Path denylist (writes only)

The agent may read any path. Writes are forbidden on:

| Path                     | Why                                                                       |
| ------------------------ | ------------------------------------------------------------------------- |
| `.git/`                  | git internals — corruption is silently disastrous                          |
| `vercel.json`            | deployment config — tied to TinaCloud rebuild hook                         |
| `tina/__generated__/`    | regenerated by `tinacms build`; hand edits overwritten                    |
| `node_modules/`          | managed by the package manager                                            |
| `.env`, `.env.local`     | secrets                                                                   |
| `package-lock.json`      | only via `pnpm install`, never hand-edited                                |
| `pnpm-workspace.yaml`    | reviewed per change                                                       |
| `tsconfig.json`          | reviewed per change                                                       |
| Any `*.min.js`, `*.min.css` | generated artifacts                                                  |
| `dist/`, `build/`, `.astro/`, `.vercel/` | build outputs                                                |

## Operation denylist

The agent must not run any of:

```bash
git push origin *                # human gate
git reset --hard                  # unrecoverable
git clean -fd                     # destroys untracked work
git branch -D *                   # unrecoverable

gh repo delete *                  # unrecoverable
gh issue close *                  # human gate
gh pr close *                     # human gate
gh issue edit --remove-label *    # human gate

vercel deploy --prod *            # human gate
tina schema push *                # does not exist; included for intent
pnpm publish *                    # irreversible
rm -rf                            # within repo root only allowed by human override with recorded path
```

## Schema denylist

The agent must not ship **breaking** TinaCMS / Astro Content Collection
schema changes without a per-pass human override recorded in
`loop-run-log.md`:

- `tina/collections/*.ts` — additive changes OK; removals require override
- `src/content/config.ts` — additive only
- Field renames — always require override (drop + add reads as a removal)

## Override flow

A human may issue a per-pass override that lifts **exactly one** denylist
item. The override is recorded in the next `loop-run-log.md` entry as:

```
Override:
  - Path / op:  <exactly as listed above>
  - Granted by: <name>
  - Rationale:  <one line>
```

If two denylist items need lifting, that is two separate passes; the loop
never auto-bundles overrides.
